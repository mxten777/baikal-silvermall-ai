## 6. 인증 및 보안

### 6.1 Supabase Auth

- **방식**: 이메일+패스워드 (소셜 로그인은 2차)
- **토큰**: JWT (Supabase 자동 발급/갱신)
- **세션**: `@supabase/ssr` 미들웨어로 Next.js App Router에서 서버 측 세션 관리

### 6.2 역할(Role) 체계

| 역할 | 설명 | 접근 가능 영역 |
|------|------|----------------|
| `user` | 일반 회원 | 스토어, 마이페이지 |
| `admin` | 관리자 | 전체 (어드민 포함) |

`users.role` 컬럼으로 관리, `middleware.ts`에서 `/admin/**` 진입 시 role 검증.

### 6.3 RLS (Row Level Security)

모든 테이블에 RLS 적용:

```sql
-- 예시: orders 테이블
USING (user_id = auth.uid())              -- 조회: 본인 주문만
WITH CHECK (user_id = auth.uid())         -- 삽입/수정: 본인만
-- admin은 별도 policy로 전체 접근 허용
```

### 6.4 Edge Function 보안

- Toss Payments Secret Key는 Supabase Secret으로만 보관
- 웹훅 HMAC 서명 검증 필수
- `SUPABASE_SERVICE_ROLE_KEY` 는 Edge Function 내부에서만 사용

## 7. 결제 흐름 (Toss Payments)

### 7.1 결제 준비 (`POST /api/checkout/prepare`)

```typescript
// 1. 주문 생성 (status: pending_payment)
// 2. 재고 임시 확보 (선택)
// 3. 결제 고유키(orderId), 금액 반환
```

### 7.2 결제 승인 (`Edge Function: payments-toss-confirm`)

```typescript
// 1. Toss API POST /v1/payments/confirm
// 2. 트랜잭션 처리:
//    - payments.status = 'done'
//    - orders.status = 'paid', paid_at = NOW()
//    - inventory qty 차감
//    - points_ledger INSERT (적립 예약)
//    - user_coupons.used_at UPDATE
//    - notifications INSERT
```

### 7.3 웹훅 (`Edge Function: webhooks-toss`)

- 이벤트: `PAYMENT_STATUS_CHANGED`, `VIRTUAL_ACCOUNT_DEPOSIT`
- HMAC-SHA256 서명 검증 후 멱등 상태 업데이트

## 8. API 설계

### 8.1 Route Handlers (Next.js)

| 경로 | 메서드 | 설명 |
|------|--------|------|
| `/api/checkout/prepare` | POST | 주문 생성·결제 준비 |
| `/api/checkout/confirm` | POST | 결제 승인 (→ Edge Function 호출) |
| `/api/recommendations` | GET | 상품 추천 (프로필 기반) |

### 8.2 Edge Functions (Supabase)

| 함수명 | 트리거 | 설명 |
|--------|--------|------|
| `payments-toss-confirm` | HTTP POST | Toss 결제 승인 처리 |
| `webhooks-toss` | HTTP POST | Toss 웹훅 수신 |
| `admin-orders-ship` | HTTP POST | 주문 배송 처리 (송장 등록) |
